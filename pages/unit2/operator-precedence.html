Operator Precedence

Why a + b * 2 doesn't mean "add first, then multiply" — the order in which C evaluates mixed operators within a single expression.

What Is It?

Operator precedence determines the order in which different operators are evaluated within a single expression when there's no explicit grouping. Just like standard math — "multiplication before addition" — C follows a defined precedence order: for example, * and / are evaluated before + and -.

Why Do We Need to Know This?

Misunderstanding precedence is a very common source of subtly wrong calculations. A program can compile perfectly and still produce the wrong numeric result if you assumed a different evaluation order than C actually uses.

Real-Life Example

Think of the standard math classroom rule: "multiply and divide before you add and subtract." C follows a similar, well-defined set of rules for its own operators — just with more operator categories involved.

Syntax

Operator precedence has no special syntax of its own — it's a rule about evaluation order, not a symbol you write. Parentheses () can always be used to override the default precedence and force a specific evaluation order explicitly.

syntax 
a + b * 2      // * evaluated first (higher precedence), then +
(a + b) * 2    // parentheses force + to be evaluated first instead

Step-by-Step Code

Extending add.c with an expression that mixes multiplication and addition:

add.c 
#include <stdio.h>

int main() {
    int a = 5;
    int b = 3;
    int result = a + b * 2;

    printf("Result: %d\n", result);

    return 0;
}
  • Even though it's written left-to-right as a + b * 2, the * (multiplication) has higher precedence than + (addition).
  • So b * 2 is calculated first: 3 * 2 = 6.
  • Only then is a added to that result: 5 + 6 = 11 — not (a + b) * 2, which would incorrectly give 16.

Dry Run

Tracing through execution with a = 5, b = 3:

  • b * 2 is evaluated first, because * has higher precedence than +: 3 * 2 = 6.
  • Then a + 6 is evaluated: 5 + 6 = 11.
  • result is assigned 11.

The exact output is:

output 
Result: 11

Contrast this with the wrong assumption of strict left-to-right evaluation, which would incorrectly suggest (5 + 3) * 2 = 16. The actual result is 11, not 16 — precedence rules, not pure left-to-right reading order, determine which operator runs first.

Common Mistakes

  • Assuming expressions are always evaluated strictly left-to-right regardless of operator type — they're not. Precedence rules take priority over pure left-to-right reading.
  • Forgetting that relational and logical operators also have their own precedence relative to arithmetic operators — arithmetic generally happens before relational comparisons, which happen before logical combinations. It's a nuance worth being aware of even without memorizing every rule.

Debugging Tips

If a calculated result doesn't match your expectation, mentally (or on paper) apply "multiplication/division before addition/subtraction" first, and consider adding explicit parentheses to test whether your assumed grouping was actually correct.

Best Practices

  • Use parentheses liberally to make your intended evaluation order explicit and unambiguous, even in cases where the default precedence would technically already produce the correct result — it makes code easier for others (and future you) to read correctly at a glance.

FAQs

Q: Do I need to memorize the full precedence table for every operator in C?
A: Not entirely — knowing the most common cases, like multiplication/division before addition/subtraction, and using parentheses when unsure, covers the vast majority of real situations.

Q: Is precedence the same thing as associativity?
A: Related but distinct — precedence determines which operator type goes first; associativity (covered next) determines the order among operators of the same precedence level.

Key Takeaways

  • Operator precedence determines the order in which different operators are evaluated within an expression when there's no explicit grouping.
  • The most important everyday case: multiplication and division are evaluated before addition and subtraction.
  • Parentheses are the reliable way to make your intended evaluation order explicit, regardless of default precedence rules.
