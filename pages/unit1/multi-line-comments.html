Multi Line Comments

The last building block of Unit 1 — a comment style that can stretch across as many lines as you need, bracketed off by /* and */.

1. What Is It?

A multi-line comment starts with /* and ends with */. Everything in between — even across several lines — is ignored by the compiler.

2. Why Do We Need to Know This?

Multi-line comments are useful for longer explanations, for temporarily disabling multiple lines of code at once, or for writing a header block that describes an entire file or function.

3. Real-Life Example

Like bracketing off an entire paragraph in a document with a note that says "skip this section" — everything inside the brackets is set aside, no matter how many lines it spans.

4. Visual Explanation

example.c 
/*
   This program demonstrates a simple
   addition and prints the result.
*/

This comment spans 3 lines, all of which are ignored by the compiler. It only ends when the compiler reaches */ — not at the end of any individual line.

5. Syntax

  • /* marks the start of the comment.
  • */ marks the end of the comment.
  • Everything between them — including line breaks — is part of the comment.
  • Unlike //, a multi-line comment does not end automatically at a line break; it only ends at */, however many lines later that may be.

6. Step-by-Step Code

Here is add.c again, this time with a multi-line header comment added on top of its existing single-line comments:

add.c 
/*
   Program: add.c
   Purpose: Prints the sum of two fixed numbers (5 and 3)
   as part of this course's Unit 1 pipeline chapter.
*/
#include <stdio.h>  // Needed for printf

int main() {
    // Print the result of adding 5 and 3
    printf("The sum of 5 and 3 is 8\n");
    return 0;
}

The multi-line comment at the top acts as a file header, describing the whole program at a glance. Meanwhile, the // comments from before still work exactly as before — both comment styles can coexist happily in the same file.

7. Common Mistakes

  • Forgetting the closing */ — this accidentally comments out all the rest of the file until the compiler finds a */ somewhere, which causes confusing errors that can be hard to trace back.
  • Trying to nest one multi-line comment inside another, like /* outer /* inner */ still outer */. This does not work as expected in C — the first */ closes the whole comment, leaving still outer */ as stray code, which causes errors.

8. Debugging Tips

Wrapping a whole block of code in /* ... */ is a fast way to temporarily disable several lines at once during debugging, without deleting them or commenting out each line individually with //.

9. Best Practices

  • Always double-check that you've closed a multi-line comment with */ before moving on.
  • Use multi-line comments for file/function header blocks or for temporarily disabling multi-line blocks of code, and use // for short inline notes.

15. FAQs

Q: Can I mix // and /* */ comments in the same file?
A: Yes, freely — both styles coexist normally in the same C file with no conflict.

Q: Which comment style should I default to?
A: Use // for short inline notes, and /* */ for longer explanations or header blocks — both are equally valid C.

16. Key Takeaways

  • A multi-line comment starts with /* and ends with */, with everything in between ignored by the compiler.
  • It does not end automatically at a line break — only */ ends it, so it can span as many lines as needed, and one cannot be nested inside another.
