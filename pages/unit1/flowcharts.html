Flowcharts

The third and final way to represent an algorithm in this chapter: as a picture, drawn with shapes and arrows instead of written words.

1. What Is It?

A flowchart is a diagram that represents an algorithm's steps as connected shapes: ovals for start/end, rectangles for actions/steps, diamonds for decisions, and arrows for the direction of flow. It's the visual counterpart to pseudocode — the exact same logic, drawn instead of written.

2. Why Do We Need to Know This?

Some people, and some problems, are simply easier to reason about visually than as text. Flowcharts also make it trivial to spot missing branches or dead ends in logic — a decision diamond with only one arrow leaving it is an obvious, visible bug that might hide for longer in a page of written pseudocode.

3. Real-Life Example

A subway or metro map is a flowchart-like diagram: stations are steps, junctions are decisions ("change lines here"), and the colored lines are arrows showing every possible route — all understandable at a glance, without reading a paragraph of written directions.

5. Flowchart Symbol Rules

  • Exactly one Start and one End shape per flowchart — never more, never fewer.
  • Every decision diamond must have exactly two labeled arrows leaving it — commonly "Yes" and "No".
  • Arrows only flow in the direction of execution — avoid unlabeled loops back to earlier boxes without a clear, obvious reason.

10. Common Mistakes

  • Missing one branch of a decision diamond (e.g. only drawing what happens if "Yes," with no arrow for "No").
  • Drawing multiple "Start" boxes, or forgetting to include an "End" box at all.
  • Arrows that don't clearly show direction, leaving the reader unsure which way the flow moves.
  • Cramming more than one action into a single box — each box should represent exactly one step, not several.

11. Debugging Tips

If a flowchart "looks right" but the algorithm it represents still fails, check every decision diamond and confirm both branches actually lead somewhere — a dangling branch (an arrow that trails off, or a "No" path that was simply never drawn) is the most common flowchart-logic bug, and it hides in plain sight until you deliberately trace both arrows out of every diamond.

12. Best Practices

  • One action per box — if you catch yourself writing "and" inside a rectangle, split it into two boxes.
  • Label every decision branch explicitly — "Yes"/"No", or the actual condition outcome, never leave a branch unlabeled.
  • Keep the whole flowchart readable in one consistent direction — top-to-bottom or left-to-right — not zig-zagging across the page.

17. Mini Project

Plan a Real-World Process, Three Ways.

You've now met all three ways of representing an algorithm's logic taught in this chapter: as a numbered plain-English algorithm, as pseudocode, and as a flowchart. This mini project is your chance to prove you can move between all three for a process you choose yourself.

Pick a real task from everyday life — for example, making a cup of tea, checking out a library book, or crossing a busy road safely. It should have at least one decision point (something that could go two different ways), so your flowchart actually needs a decision diamond.

Produce all three of the following for your chosen task:

  • A numbered plain-English algorithm — following the conventions from the Algorithms page (clear, ordered, unambiguous steps).
  • The same logic as pseudocode — using the keywords from the Pseudocode page (INPUT, IF/ELSE, DISPLAY, and so on).
  • A description of the same logic as a flowchart — following this page's symbol rules (one Start, one End, every decision diamond with two labeled branches).

When you're done, compare all three side by side. They should describe exactly the same underlying logic — just in three different shapes. If any of them disagree with the others, that's a sign one of your three versions has a bug worth fixing. Completing this project means you've genuinely internalized the core idea of this entire chapter: an algorithm is the logic, and plain English, pseudocode, and a flowchart are just three different costumes it can wear.

18. Chapter Summary

A flowchart is the visual representation of an algorithm's logic — the same underlying reasoning as plain-English algorithm steps or pseudocode, just drawn instead of written. Across this chapter, one recurring example — largest of two numbers — was represented three ways: numbered steps, pseudocode, and a flowchart, proving they're three views of the exact same idea.

19. Cheat Sheet

  • Oval — Start / End.
  • Rectangle — Process (an action or step).
  • Diamond — Decision.
  • Arrow — Flow direction.
  • Hard rule: every decision needs two labeled branches.

20. FAQs

Q: Do I need special software to draw flowcharts?
A: No — pen and paper, or any plain diagram, is enough. This course draws flowcharts in HTML/CSS for the web page, but drawing them by hand is just as valid.

Q: Which is "better," pseudocode or a flowchart?
A: Neither — pick whichever helps you reason about a specific problem. Some people, and some problems, suit one better than the other.

21. Key Takeaways

  • Flowcharts are algorithms drawn as shapes — ovals, rectangles, diamonds, and arrows.
  • Every decision needs two labeled branches; a dangling or missing branch is a visible bug.
  • Algorithm, pseudocode, and flowchart are three representations of the same logic — this chapter is complete once you can move fluently between all three.
