Source Code

The very first artifact in the compilation pipeline — the human-readable text a programmer writes, before any tool has touched it.

1. What Is It?

Source code is the human-readable text a programmer writes — like add.c — following a programming language's rules (its syntax). It's the very first artifact in the compilation pipeline, and the only one a person is expected to read and write directly. Everything that comes later — object code, executables — is derived from it by tools, not typed by hand.

2. Why Do We Need to Know This?

Recognizing what makes something "source code" — versus object code or an executable — clarifies exactly which files in a project you should be editing, and which ones are just compiler output you shouldn't touch by hand. Without this distinction, it's easy to waste time editing a generated file that gets silently overwritten the next time you compile.

3. Real-Life Example

Think of source code like a handwritten recipe before it's ever cooked — readable, editable, and meaningful to a person, but not yet a finished dish. You can study it, tweak an ingredient, or hand it to someone else to read. The finished dish (the executable) is what you get only after following the recipe through a process — in this case, compilation.

4. Visual Explanation

Here is this chapter's recurring example, add.c, shown once more as pure source code:

Source Code — add.c 
#include <stdio.h>

int main() {
    printf("The sum of 5 and 3 is 8\n");
    return 0;
}

Notice the .c file extension — it signals to every tool involved that this file contains C source code. Every character inside it is meant for both the compiler and a human reader: the compiler will translate it into machine instructions, while you (or anyone else) can open it in a text editor and understand exactly what it's supposed to do.

5. Common Mistakes

  • Assuming source code and object code are interchangeable terms — they are two very different stages of the same pipeline.
  • Forgetting that source code must be saved with the correct file extension (.c for C) for tools to recognize it properly.
  • Editing generated or compiled files instead of the original source — any changes made there would be lost on the next recompile.

6. Debugging Tips

If something about your program's behavior seems wrong, the fix always happens in the source code — never in the compiled output. Compiled files are regenerated from source every single time you compile, so any change you make directly to an object file or executable is invisible to the next build and will simply vanish. When in doubt about where to make a fix, the answer is always: open the .c file.

7. Best Practices

  • Keep source code organized in clearly named files.
  • Use the standard .c extension so tools and other developers immediately recognize the file type.

12. FAQs

Q: Can source code be written in any text editor?
A: Yes — source code is plain text, so any text editor works, though code-aware editors add helpful features like syntax highlighting and error checking.

Q: Is source code the same as a "script"?
A: Related but not identical — "script" often implies interpreted code run directly, while C source code must be compiled first before it can run.

13. Key Takeaways

  • Source code is the human-readable, human-written starting point of the compilation pipeline.
  • It must be compiled before it can run — a computer cannot execute source code directly.
  • Always edit the source, never the compiled output.
