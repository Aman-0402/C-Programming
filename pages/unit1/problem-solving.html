Problem Solving

Before you write a single line of C, you need a plan. Coding is the last step of problem solving — not the first.

1. What Is It?

Problem solving in programming means figuring out what you need to do and how before you touch a keyboard to write code. It follows a repeatable four-stage framework:

  1. Understand — read the problem carefully and be sure you know exactly what is being asked, what inputs you have, and what output is expected.
  2. Plan — decide on a general approach or strategy for solving it, without worrying about syntax yet.
  3. Divide — break the problem into smaller, more manageable sub-problems.
  4. Solve — work out a solution for each sub-problem, one at a time.

Only after all four stages are done does writing actual code make sense. Coding is where you translate a plan into a language the computer understands — it is not where thinking happens.

2. Why Do We Need to Know This?

The single most common failure for beginners is not a syntax error — it's opening an editor and typing code before they actually understand the problem. This leads to programs that are half-built, contradict themselves, or solve the wrong problem entirely. Learning to pause and work through Understand → Plan → Divide → Solve before coding prevents the single biggest source of beginner frustration.

3. Real-Life Example

Think about planning a trip. You don't start packing a suitcase before you've decided where you're going. The natural order is: understand the goal (where and why you're traveling), plan the approach (budget, dates, mode of transport), then break the plan into smaller tasks — book transport, arrange accommodation, pack, plan a route for each day — and finally do each one. Skipping straight to "packing" without a destination or budget wastes time and often produces a trip that doesn't actually work.

10. Common Mistakes

  • Jumping straight to writing code before understanding the problem.
  • Trying to solve a big problem all at once instead of breaking it into smaller ones.
  • Assuming the first idea that comes to mind is the only possible approach.

11. Debugging Tips

When you get stuck, don't stare harder at code that doesn't exist yet. Go back one stage instead: re-read the problem statement first, not any code or notes you've already written. Most "stuck" moments are actually a sign that the Understand or Divide stage was rushed, not that the sub-problem itself is unsolvable.

12. Best Practices

  • Restate the problem in your own words before attempting to solve it — if you can't, you don't understand it yet.
  • Write down the inputs you're given and the exact output you expect before attempting a solution.
  • If a problem feels too big, solve a smaller or simpler version of it first, then build up from there.

15. Programming Exercises

Easy: Write out the four stages of problem solving (Understand, Plan, Divide, Solve) from memory, with a one-sentence description of each.

Medium: Break "plan a birthday party" into at least 5 sub-problems.

Hard: Take the problem "find the largest number in a list of 10 numbers" and write out Understand / Plan / Divide / Solve for it in plain English — no code needed. (You'll turn this into actual logic in the Sequential Logic and Algorithms topics.)

16. Assignments

Pick any everyday task — making a sandwich, doing laundry, packing a school bag — and write it out fully using the four-stage framework: Understand, Plan, Divide, Solve.

18. Chapter Summary

Problem solving is a repeatable four-stage process — Understand, Plan, Divide, Solve — that should happen before any code is written. Building this habit early is what separates programmers who work through problems confidently from those who get stuck the moment they open an editor.

19. Cheat Sheet

  • Understand — know exactly what's being asked, the inputs, and the expected output.
  • Plan — decide on a general approach before worrying about syntax.
  • Divide — break the problem into smaller, manageable sub-problems.
  • Solve — work out each sub-problem one at a time.

20. FAQs

Q: Do I have to follow these 4 stages formally every time?
A: Not rigidly — but internalizing the habit prevents most beginner mistakes, even once it becomes automatic and you're not writing each stage down.

Q: What if I can't break the problem down?
A: Try solving a smaller or simpler version of the problem first, then build up from there.

21. Key Takeaways

  • Problem solving follows a repeatable four-stage framework: Understand, Plan, Divide, Solve.
  • Decomposition — breaking a big problem into smaller sub-problems — makes even large problems manageable.
  • Coding is the last step of problem solving, not the first.
