Single Line Comments

A single-line comment starts with // and extends to the end of that one line — the most common way to add a quick note right next to the code it describes.

1. What Is It?

A single-line comment starts with // and extends to the end of that line only — everything after // on that line is ignored by the compiler. It has no effect on the code that runs; it exists purely for human readers.

2. Why Do We Need to Know This?

It's the most common comment style for short, quick explanations placed right next to the specific line they describe. Instead of scrolling to a separate note, the explanation sits exactly where it's needed.

3. Real-Life Example

Like a margin note scribbled next to one specific sentence in a book — it applies only to that one line, not the whole page. You wouldn't expect a note next to line 3 to also explain line 4.

4. Visual Explanation

example.c 
int total = 8; // This will hold the final sum

Everything from // to the end of the line is a comment, and it is ignored by the compiler. But int total = 8; is still real code that runs — the comment only describes it, it doesn't disable it.

5. Syntax

// marks the start of a comment. Everything after it, on that same line, is ignored by the compiler. The comment automatically ends at the line break — there is no closing symbol needed, unlike multi-line comments.

6. Step-by-Step Code

Here is add.c with single-line comments added at two points:

add.c 
#include <stdio.h>  // Needed for printf

int main() {
    // Print the result of adding 5 and 3
    printf("The sum of 5 and 3 is 8\n");
    return 0;
}

Two different placements are shown here. The comment after #include <stdio.h> is a trailing comment — it sits at the end of a code line, explaining that line directly. The comment above printf(...) is a full-line comment — it sits on its own line, explaining the statement that follows it. Both are valid, common uses of single-line comments.

7. Common Mistakes

  • Forgetting that // comments only extend to the end of that one line — accidentally assuming it comments out multiple lines below it, when it does not.
  • Putting code after // on the same line, expecting it to still run — it won't. Everything after // on that line is ignored, including any code you put there.

8. Debugging Tips

  • You can quickly disable one specific line by placing // at its start.
  • If a program starts behaving correctly after doing this, that line is worth investigating further — it's a fast way to narrow down which line is causing a problem.

9. Best Practices

  • Keep single-line comments short and placed close to the code they describe.
  • Prefer a comment on its own line above complex code, and a short trailing comment for simple clarifications.

12. Programming Exercises

Hard: Comment out one specific line of a short program and explain what changes as a result.

13. Assignments

Take add.c and add a single-line comment to every logical section: the #include line, the main declaration, the printf call, and the return statement.

14. FAQs

Q: Can I put a single-line comment at the very start of a file?
A: Yes — a single-line comment can go anywhere a comment could go, including as the very first line of a file.

Q: Is // the same in all programming languages?
A: No — some languages use different symbols for single-line comments, like # in Python.

15. Key Takeaways

  • // marks the start of a single-line comment in C.
  • A single-line comment only extends to the end of that one line — it automatically ends at the line break with no closing symbol needed.
  • Use it for a quick trailing note at the end of a code line, or a short standalone note on its own line above the code it describes.
