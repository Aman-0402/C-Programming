Interpreter

Not every language works like C. An interpreter reads and runs source code directly, line by line — a completely different strategy from the compile-then-run pipeline you've been tracing.

What Is It?

An interpreter is a program that reads and executes source code directly, line by line, without producing a separate standalone executable file first. Instead of translating an entire program ahead of time and saving the result as an executable, an interpreter reads one line (or statement), translates it, and runs it immediately — then moves to the next line, and repeats. This is a fundamentally different strategy from C's compile-then-run approach, where translation and execution are two completely separate steps.

Why Do We Need to Know This?

You've just spent this entire chapter tracing C's compiler pipeline — preprocessing, compilation, assembly, linking, and loading — all of which happens before a program ever runs. Contrasting interpreters with that pipeline makes C's specific approach, and its speed and portability tradeoffs, much clearer by comparison. Once you see the alternative, C's design choices stop looking arbitrary and start looking deliberate.

Real-Life Example

Like a live interpreter at a conference translating a speaker's sentences one at a time, in real time, as the speech happens — versus a translator who translates an entire document in advance, before anyone reads a single word of it (the compiler). The live interpreter never produces a saved, complete translated transcript ahead of time; they translate live, every single time the speech is given. The document translator does the work once, up front, so the finished translation can be read instantly, as many times as needed, without retranslating.

Common Mistakes

  • Assuming interpreted languages don't get translated at all — they do, just differently: line by line, at run time, instead of all at once ahead of time.
  • Assuming C is interpreted — it isn't. C is a compiled language; it always goes through the full compile-then-run pipeline covered earlier in this chapter.
  • Assuming interpreters and compilers are just two names for the same thing — they represent two genuinely different strategies for getting from source code to running behavior.

Debugging Tips

Interpreted languages often report errors as soon as they hit the bad line during execution — the program may have already printed output from earlier, correct lines before it fails. Compiled languages like C, by contrast, catch many errors earlier, during compilation, before the program ever runs at all. This is useful context when comparing debugging experiences across languages: a C program that compiles has already passed a syntax check that an interpreted script simply hasn't, because that script isn't checked until each line is actually reached.

Best Practices

  • Recognize which category — compiled or interpreted — a new language falls into early, before you start writing serious code in it.
  • Knowing the category changes your expectations for speed, startup behavior, and when errors will surface.
  • Don't assume every language behaves like C; always check whether it's compiled, interpreted, or something else before comparing performance or debugging habits.

FAQs

Q: Is one approach objectively better?
A: No — each has tradeoffs. Compiled languages favor speed and catching errors early; interpreted languages favor flexibility and fast iteration during development.

Q: Can a language be both?
A: Some languages and runtimes blend both strategies, but that's beyond the scope of this course.

Key Takeaways

  • An interpreter reads and executes source code directly, line by line, without first producing a standalone executable.
  • Compiled languages translate once ahead of time and run the result many times; interpreted languages translate and run each line every time the program runs.
  • C is a compiled language, not an interpreted one.
