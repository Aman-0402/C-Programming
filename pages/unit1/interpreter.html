Interpreters, Linkers, and Loaders

Many students confuse the compiler, interpreter, linker, and loader. By the end of this lesson, you'll clearly understand the role of each.

The Big Picture

When you write a C program, it goes through several stages.

C Program (.c)
Preprocessor
Compiler
Object File (.o/.obj)
Linker
Executable File (.exe)
Loader
CPU
Program Execution

Notice that Interpreter is NOT part of the C compilation process, because C is a compiled language.

What is an Interpreter?

Definition: An Interpreter is a program that reads, translates, and executes source code one statement at a time. Unlike a compiler, it does not create a separate executable file.

Simply put: Interpreter = Translate one line → Execute it → Repeat.

Real-Life Example

Imagine an English teacher translating a speech into Hindi.

The speaker says one sentence.
Interpreter translates that sentence.
Speaker says the next sentence.
Interpreter translates it.

Everything happens one sentence at a time.

How an Interpreter Works

example.py 
print("Hello")

print("World")

Execution:

Read Line 1
Execute Line 1
Read Line 2
Execute Line 2

There is no executable file created.

Languages That Use Interpreters

Examples: Python (primarily interpreted), JavaScript, Ruby, PHP (traditionally interpreted).

Note: Many modern implementations also use compilation techniques internally (such as bytecode and JIT compilation), but from a beginner's perspective they are commonly taught as interpreted languages.

Advantages of an Interpreter

  • Easy to test code.
  • Stops immediately when it finds an error.
  • No separate compilation step.
  • Good for scripting and rapid development.

Disadvantages

  • Slower execution because translation happens during execution.
  • Source code must usually be available to run.

What is a Compiler?

You already learned this, but let's review. A Compiler translates the entire program at once and creates an executable file.

What is a Linker?

Definition: A Linker combines object files and library files into a single executable program. Without a linker, your program is incomplete.

Why Do We Need a Linker?

Suppose you write:

hello.c 
#include <stdio.h>

int main()
{
    printf("Hello");

    return 0;
}

You wrote printf(). But where is the actual code for printf()? It exists in the Standard C Library. The linker connects your program with the required library.

Linking Process

Your Object File + Standard Library
Linker
Executable File

Static Linking

In Static Linking, the required library code is copied into the executable.

Program + Library
One Large Executable

Advantages

  • Runs even if the library is not installed later.
  • Self-contained executable.

Disadvantages

  • Larger executable size.
  • If a library is updated, the executable must usually be rebuilt to use the new version.

Dynamic Linking

In Dynamic Linking, the executable contains references to shared libraries instead of copying all library code. The required library is loaded when the program starts (or when first needed).

Program
Uses Shared Library
Execution

Advantages

  • Smaller executable.
  • Multiple programs can share the same library in memory.
  • Updating a shared library can benefit multiple programs (while maintaining compatibility).

Disadvantages

  • The required shared library must be available on the system.

Static vs Dynamic Linking

Static LinkingDynamic Linking
Library code copied into executableShared library used at runtime
Larger executableSmaller executable
Independent of shared library installationDepends on shared library availability
Common in standalone toolsCommon in modern operating systems

What is a Loader?

Definition: A Loader is part of the Operating System. Its job is to:

  • Load the executable into RAM.
  • Allocate memory.
  • Prepare the program for execution.
  • Transfer control to the program's entry point (main() in a C program).

Loader Process

Executable File
RAM

main()
Program Runs

Responsibilities of the Loader

  • Loads the executable into memory.
  • Allocates memory for code, data, stack, and heap.
  • Resolves remaining runtime linking tasks (for dynamically linked programs).
  • Starts execution.

Interesting Facts 💡

  • The Loader is part of the Operating System, not the compiler.
  • Large applications may link with hundreds of shared libraries.
  • Dynamic linking reduces memory usage because multiple programs can share the same library in RAM.
  • C is a compiled language, so an interpreter is not used to execute standard C programs.

Quick Revision

  • Interpreter executes code one statement at a time.
  • Compiler translates the entire program and creates an executable.
  • Linker combines object files and libraries.
  • Loader loads the executable into memory and starts the program.
  • Static Linking copies library code into the executable.
  • Dynamic Linking uses shared libraries at runtime.

Because your program's object code may call functions, like printf(), whose actual implementation lives in the Standard C Library rather than in your own source file. The linker connects your code with that library to produce a complete executable.

Intermediate

Source code is compiled into an object file, the linker combines that object file with required libraries into an executable, the loader loads the executable into RAM and allocates memory, and finally the CPU begins executing instructions starting at main().

Static linking copies the library code directly into the executable, producing a larger but self-contained file that doesn't depend on the library being installed separately. Dynamic linking keeps references to shared libraries instead, producing a smaller executable that depends on the shared library being available at runtime, but allows multiple programs to share the same library in memory.

Because C is a compiled language — its entire program is translated ahead of time into a standalone executable by the compiler, assembler, and linker, so there's no need to translate and execute it line by line at runtime like an interpreter does.

It loads the executable into RAM, allocates memory for code, data, stack, and heap, resolves any remaining runtime linking tasks for dynamically linked programs, and then transfers control to the program's entry point, main().

Shared libraries keep executables smaller since the library code isn't duplicated inside each one, and multiple programs can share the same copy of a library in memory at once, reducing overall memory usage.
