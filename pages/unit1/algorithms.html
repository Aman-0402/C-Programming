Algorithms

An algorithm is what you get when a plan becomes precise enough for a machine to follow — the first idea in this chapter you'll carry, unchanged, into every C program you ever write.

1. What Is It?

An algorithm is a finite, precise, ordered set of steps to solve a problem or complete a task. It is the output of the "Plan" and "Divide" stages from the Problem Solving topic, made rigorous enough that a computer — not just a human — could follow it exactly, with nothing left to guesswork.

2. Why Do We Need to Know This?

Every program is an implementation of an algorithm. Learning to design an algorithm in plain steps first — before any C syntax — means the syntax (taught starting in the next chapter) only has to represent an idea you already have clearly, instead of asking you to invent the idea and the syntax at the same time.

3. Real-Life Example

A recipe is an algorithm: precise ordered steps ("preheat oven to 350°F," not "make it hot-ish") with clear inputs (ingredients) and a clear output (the finished dish). Skip a step, or do them out of order, and the recipe fails the same way a program fails when its algorithm is wrong.

5. Algorithm-Writing Conventions

Algorithms in this course follow a few simple conventions so they stay unambiguous and easy to translate into code later:

  • Steps are numbered in order: Step 1, Step 2, Step 3…
  • Every algorithm begins with a Start marker and ends with an End marker.
  • Each step describes exactly one action — never bundle two actions into one step.
  • Decisions are written as "If [condition], then… else…" so every branch is explicit — never leave an "else" implied.

6. Step-by-Step Code

algorithm.txt 
Step 1: Start
Step 2: Input two numbers, A and B
Step 3: If A > B, go to Step 4, else go to Step 5
Step 4: Display A as the largest, go to Step 6
Step 5: Display B as the largest
Step 6: End

Step 1 and Step 6 mark where the algorithm begins and ends — every algorithm needs both, so it's never left "hanging" with an undefined finish. Step 2 gathers the raw inputs before any decision is made. Step 3 is the one decision in this algorithm, written with both an explicit "yes" path (Step 4) and "no" path (Step 5) — leaving either branch out would make the algorithm incomplete. Steps 4 and 5 each display exactly one result, and Step 4 explicitly jumps past Step 5 so both outcomes can never run together.

7. Dry Run

A dry run traces an algorithm by hand, step by step, using real numbers — the fastest way to catch a logic mistake before it becomes a bug in real code.

8. Common Mistakes

  • Writing vague, unmeasurable steps like "make it work" instead of a precise, checkable action.
  • Missing an end condition, leaving the algorithm infinite or undefined about when it should stop.
  • Skipping the decision's "else" branch, so one possible outcome is never accounted for.

9. Debugging Tips

If an algorithm gives the wrong answer, dry-run it by hand with a simple example first, exactly like the table above. Most "logic bugs" — decisions with a missing branch, a step in the wrong order, a comparison flipped the wrong way — are visible without ever writing a single line of real code.

10. Best Practices

  • Keep each step small and singular — one action per step.
  • Always account for every branch of a decision, including the "else" case.
  • Test your algorithm by hand with at least 2 different inputs before trusting it.

13. Programming Exercises

Easy: Write an algorithm (plain steps) to check if a number is even or odd.

Medium: Write an algorithm to find the largest of THREE numbers.

Hard: Write an algorithm to check whether a given year is a leap year. Use the real divisibility rule: a year is a leap year if it is divisible by 4, but not by 100 — unless it is also divisible by 400.

15. FAQs

Q: Is an algorithm the same as code?
A: No — code is one specific implementation of an algorithm in a specific language.

Q: Can one problem have multiple correct algorithms?
A: Yes.

16. Key Takeaways

  • Algorithms are precise, finite, and ordered.
  • Dry-running by hand catches logic errors early.
  • An algorithm is language-independent; code is the C-specific implementation of it.
